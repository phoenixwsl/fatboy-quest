# Principles — 色彩、形状、排版的深度原理

读这份文件，当你需要理解 SKILL.md 简版规则**为什么**这么定时。每条原则附原作出处 + UI 落地翻译。

---

## 一、Gurney 色光学（《Color and Light: A Guide for the Realist Painter》）

James Gurney 是当代北美最重要的具象绘画教学者之一。他的"光与色"体系是 Pixar、迪士尼概念美术的圣经。核心 5 条：

### 1.1 阴影色温反相律

> "If light is warm, shadow is cool. If light is cool, shadow is warm." —— Color and Light, p. 132

**绘画原理**：太阳是暖光时，阴影里照到的是天光（冷蓝）；阴天冷光下，阴影反而暖。这是物理的，不是审美选择。

**UI 翻译**：
- 暖色主题的容器底色用奶白 `#FFF8EC`，**卡片阴影必须偏冷蓝** `rgba(60, 80, 120, 0.12)`，而不是黑色阴影 `rgba(0,0,0,0.12)`。
- 反过来：星空蓝深底，"亮起"的元素（按下后回弹）必须**带暖色高光** `inset 0 1px 0 rgba(255, 220, 180, 0.15)`。

### 1.2 灰色用补色调和

> "Pure gray is a missed opportunity." —— Gurney 教学视频

**UI 翻译**：`--ink-muted` 不用 `#888`。用主色和它的补色按 7:3 混合得到的"有色灰"：
- 暖橙主题：`oklch(60% 0.04 30deg)`（带 5% 暖色的灰）
- 紫蓝主题：`oklch(70% 0.04 250deg)`（带 5% 紫蓝的灰）

### 1.3 统一光源 / color unity

> "Everything in a painting must look like it's lit by the same light." —— p. 84

**UI 翻译**：一个主题里所有 token（包括语义色）的 hue 都偏移 5-10% 向主色靠拢。冷主题里的红色比标准红更冷紫，暖主题里的红更橙。这让"主题感"渗透到每个像素。

### 1.4 饱和度梯度

**绘画原理**：表现纵深不靠明度变化，靠饱和度衰减。远山的颜色比近山更"灰"，不是更"暗"。

**UI 翻译**：层级用饱和度，不用明度。
- 主 CTA：`oklch(70% 0.18 30deg)`（满饱和）
- 次 CTA：`oklch(70% 0.10 30deg)`（半饱和）
- 三级：`oklch(70% 0.04 30deg)`（近灰）
- 三个都同 L\* 同 hue，只调 chroma — 这是 Linear / Things 的秘诀。

### 1.5 补色阴影 = 张力

**UI 翻译**：CTA 按钮的阴影色用主色的对角补色。
- 绿色按钮：阴影 `rgba(220, 80, 130, 0.18)`（紫红）
- 橙色按钮：阴影 `rgba(80, 130, 220, 0.18)`（青蓝）

这是 Duolingo 按钮"立体感"远胜其他家的关键 — 不是阴影更深，而是阴影换色相。

## 二、Albers 色彩相互作用（《Interaction of Color》）

Joseph Albers 包豪斯大师。他的核心论点：**颜色没有"自身的样子"，只有"在某个上下文里的样子"**。

### 2.1 同色异境（同色在不同底色上看着不同）

**实验**：把同一个浅灰方块放在黑底和白底上，黑底上看着发亮，白底上看着发暗。**视觉感知是相对的**。

**UI 翻译**：
- 一个主题里"看起来 OK"的 `--ink-muted` 复用到另一个主题就糊掉 — 必须按主题校准。
- 这就是 cozy 主题用 `#6B7B8C` 在浅蓝底上变得糊的根因（截图里"今日时间轴"那行）。修法是 starry 用偏冷的 muted，cozy 用偏暖的 muted，**不复用 hex**。

### 2.2 同时对比 simultaneous contrast

**实验**：灰方块放进黄底，灰会染上紫（互补色）；放进蓝底，灰会染上橙。

**UI 翻译**：
- 浅蓝大背景上，所有"中性灰"会自动染暖 — 你必须**主动让灰偏冷**才能视觉上看着"真灰"。
- 同理，紫色背景上，所有"白色"会染绿 — `--ink: #FFFFFF` 在紫底上会看着发绿，要主动偏一点冷蓝才看着真白。这就是为什么 starry 主题白 = `#F5F8FF`（偏冷）。

### 2.3 色彩相对性 → 系统化语义

**结论**：同一个红，在白底是"警告色"，在黑底就是"主色高亮色"。语义不在颜色里，在上下文里。

**UI 翻译**：`--danger` 不能跨主题复用。每个主题分别定义"在我这个上下文里读得出来'警告'的红"：
- cozy 浅底：`#E25555`（饱和高、对比强）
- starry 深底：`#FF6BD6`（亮粉，深底上才看得见）
- mecha 深底：`#FB7185`（玫红，与霓虹青反差大）

### 2.4 少即多

> "Color reveals itself through limitation." —— Interaction of Color, lesson 12

**UI 翻译**：3 色搭配比 7 色更有秩序。每个主题"敢出现的色"控制在 5 个以内（主、辅、+3 状态）。

## 三、Itten 七大对比（《The Art of Color》）

Johannes Itten 包豪斯色彩理论奠基人。他归纳了 7 种"色彩对比"，配色法则的母法则。

| 对比类型 | UI 应用 |
|---|---|
| **1. 色相对比** | 主辅色取色环上分散位置（≥ 90°） |
| **2. 明暗对比** | 文字 vs 背景 ≥ 7:1（不是 4.5:1） |
| **3. 冷暖对比** | 整主题选一个温度方向 |
| **4. 补色对比** | CTA 按钮取主题色互补色 |
| **5. 同时对比** | 见 Albers 2.2 |
| **6. 饱和度对比** | 见 Gurney 1.4 |
| **7. 面积对比** | 强色面积越小越提神 |

### 3.1 面积对比的 Itten 比例

Itten 给出过经验比例 — 强色和弱色的面积配比：

- 黄 : 紫 = 3 : 9（紫色稳重，需要 3 倍面积压住黄的张扬）
- 橙 : 蓝 = 4 : 8（暖橙比冷蓝抢眼）
- 红 : 绿 = 6 : 6（红绿等价）

**UI 翻译**：
- 紫色背景 + 黄色按钮（如 starry 主题）— 按钮面积要小（占屏幕 < 5%），周围空白多。
- 蓝色页面 + 橙色 CTA — 按钮可以稍大（占 8-10%）。
- 红绿同屏（如已评分 vs 未评分） — 等面积，相互平衡。

### 3.2 冷暖三分

Itten 把色环分冷暖两区，**最暖 = 橙、最冷 = 蓝绿**。设计三套主题时：

- 一暖（cozy）：橙黄系
- 一冷（mecha）：蓝绿系
- 一深邃（starry）：紫蓝 — 紫是冷暖中间色，是夜空的颜色

三主题分别落在色环 30° / 170° / 250° — 两两 ≥ 90° 间隔，最大化区分。

## 四、形状语言原理

形状传达情绪不输颜色，但常被忽略。三类形状语言：

### 4.1 圆角 → 友善 / 安全
- 极大圆角（24-48px）→ 童话、糖果、贴纸
- 中圆角（14-20px）→ 现代、专业、可信
- 小圆角（4-10px）→ 工具、效率、科技
- 0px 直角 → 工业、严肃、文档

**儿童产品永远 ≥ 12px**，最好 20px+。

### 4.2 形状重量 → 视觉质量
- 圆形 / 椭圆 → 最轻、最友善（Q弹）
- 圆角矩形 → 中性、可信
- 矩形 → 重、稳、严肃
- 三角 / 多边形 → 紧张、警示、能量

**CTA 按钮**：cozy 用胶囊（圆形派生）、starry 用圆角矩形、mecha 可以加切角（八边形派生）。

### 4.3 边框 → 划分秩序
- 无边框 → 流动、开放、现代
- 软边框（1px + 主色 20% 饱和）→ 划分但不打断
- 硬边框（2-3px + 高饱和）→ 强调、可点击
- 内发光边框 → 科技、霓虹

## 五、字体节奏

不展开太多 — SKILL.md 已说了重点。但补 3 点深度：

### 5.1 字体的"温度"

字体本身有温度：
- 圆头 sans（Nunito / FOT-Rodin / 站酷快乐体） → 暖
- 几何 sans（Inter / Helvetica / 思源黑体） → 中性偏冷
- 衬线（Source Serif / 思源宋体） → 暖（但偏正式）
- 等宽（JetBrains Mono / Fira Code） → 冷（科技）

**主题字体应该和主题温度匹配**：cozy 用圆头 sans，mecha 可以用等宽 / 半几何 sans。

### 5.2 主标题的"压扁"或"拉长"

文字本身的字形比例也是设计语言：
- 标题压扁 (font-stretch: 80%) → 童趣（Kirby 的所有标题都压扁）
- 标题拉长 (font-stretch: 120%) → 优雅 / 时尚
- 标题等宽 → 严肃 / 工程

### 5.3 数字字体单独定

Fredoka / Quicksand 这类**数字感比中文感强**的字体，专门给数字（积分、计时器）用。中文用 PingFang，数字用 Fredoka — 这是中文 UI 高级感的关键之一。

## 六、为什么 OKLCH 比 HEX 高级？

**HEX / sRGB 的问题**：色彩空间不是"知觉均匀"的。同样调亮 20%，绿色看着亮 30%，紫色看着只亮 10%。所以 hex 定义的主题在不同色相下感觉的"亮度"不一致。

**OKLCH 的解决**：L（lightness）是"知觉明度"，C（chroma）是"知觉饱和"，H（hue）是色相。**L 一样的两个色，看起来一样亮**。

**实战做法**：定义主色用 OKLCH，每个 token 也用 OKLCH：
```css
--primary:        oklch(70% 0.18 30deg);
--primary-soft:   oklch(95% 0.04 30deg);
--primary-strong: oklch(40% 0.20 30deg);
```

切主题时只改 hue：30 → 250 → 170 — 三套主题"亮度感"完全一致。

iOS Safari 16.4+ 支持 oklch()。可用 PostCSS / Tailwind 4 把 oklch 编译成 fallback hex。
